== Overview

Capability tokens are a class of provider token that carry hardware, system trait and boot mechanism facts through the build engine. They allow layers and build configs to express characteristic-aware logic such as selecting crypto cipher based on acceleration, conditionally pulling in firmware layers, or gating a layer entirely on a hardware requirement - without enumerating specific device classes.

Capability tokens flow through the same provider mechanism as plain label tokens (`X-Env-Layer-Provides`, `X-Env-Layer-RequiresProvider`) and are evaluated by the same `has()` function used in trigger `when=` clauses, conflict expressions, and conditional layer requires.

== Token Taxonomy

Provider tokens fall into two classes, distinguished by the presence of a colon.

**Label tokens** - plain identifiers with no colon (`systemd`, `debian-base`, `rpi-device`, `network-activator`). Arbitrary strings with no registry involvement. Any layer may declare a label token, and a conflict between two layers declaring the same label is always an error.

**Capability tokens** - colon-separated, namespaced identifiers (`hw:bluetooth`, `boot:gpt`, `hw:soc:bcm2712`). Must be defined in the capability registry, or declared by the build config file. Declaring a capability token triggers automatic expansion, hierarchical parent inference and `implies:` resolution (see <<_capability_expansion>>).

The colon is the sole discriminator. The engine uses its presence to decide whether to validate against the registry and whether to perform expansion. Capability token names are deliberately shell incompatible - colons are not valid in shell variable names. This makes it impossible to reference them in hooks or scripts. This is intentional. Do not work around it by replacing colons with underscores.

== Namespaces

The capability registry will grow over time. Its current primary use is to describe hardware and boot facts, such as:

`hw:*`:: Hardware presence facts - silicon capabilities, onboard peripherals, accessible interfaces. A fact about what the hardware is and has, independent of whether any software in the build uses it.

`boot:*`:: Boot mechanism facts - how the device boots, what the boot chain requires.

[[_capability_expansion]]
== Capability Expansion

Declaring a capability token triggers two automatic expansions, both resolved against the registry.

=== Hierarchical parent inference

Declaring `hw:X:Y` automatically also declares `hw:X`. Declaring `hw:wlan:broadcom` therefore also declares `hw:wlan`. Layer authors can gate on either the generic token (`has('hw:wlan')`) or the vendor-specific one (`has('hw:wlan:broadcom')`). Inferred parents are associated with the same declaring layer as the child.

=== `implies:` expansion

A registry entry may carry an `implies:` list of additional tokens to inject when that token is declared. This handles cross-feature associations that the hierarchy cannot capture. For example, declaring `hw:soc:bcm2712` cascades into `hw:arch:armv8`, `hw:crypto:aes-accel`, and `hw:crypto:sha-accel` - all facts derivable from the SoC identity without the layer author enumerating them. Implied tokens are themselves subject to hierarchical parent inference.

`hw:pcie` is declared by device capability and not implied by SoC. This is because the same SoC may appear on boards that expose PCIe to the user and on boards that do not. Typically, interface and peripheral accessibility belongs on device capabilities whereas SoC `implies:` covers silicon-intrinsic capabilities only.

== The Capability Registry

The registry is a set of YAML files scanned from `capability/*.yaml`. The built-in capability root is scanned first followed by the user's SRCROOT (if supplied). Within each directory, files are loaded in lexical order. Standard tokens are therefore always present before any user-defined tokens allowing them to be referenced. Attempting to override a built-in token will result in an error. Attempting to reference a token that has not been declared will also result in an error.

=== Registry format

[source,yaml]
----
hw:soc:bcm2712:
  desc: Broadcom BCM2712 SoC — Raspberry Pi 5 / CM5 (quad Cortex-A76)
  implies:
    - hw:arch:armv8
    - hw:crypto:aes-accel
    - hw:crypto:sha-accel

hw:arch:armv8:
  desc: ARMv8-A architecture
  implies:
    - hw:arch:aarch64

hw:arch:aarch64:
  desc: AArch64 64-bit execution state (ARMv8-A)

hw:bluetooth:
  desc: Onboard Bluetooth hardware — HCI interface available to the kernel
----

`desc` is required. `implies:` is optional. All `implies:` targets must themselves be defined in the registry before being referenced.

`include:` directives resolve relative to the including file and are useful for organising a large OEM token tree across subdirectories:

[source,yaml]
----
include: hw/hw.yaml
----

=== OEM custom tokens

An OEM adds a `capability/*.yaml` file to their SRCROOT. Standard built-in tokens are already loaded - no explicit include of the built-in registry is needed:

[source,yaml]
----
# SRCROOT/capability/myboard.yaml

hw:acme:
  desc: Custom board capability definitions

hw:acme:can:
  desc: CAN bus controller on this carrier board

hw:acme:gps:
  desc: GPS modem on this carrier board

hw:acme:carrierv1:
  desc: v1 carrier board with Compute Module 5
  implies:
    - hw:acme:can
    - hw:acme:gps
    - hw:device:rpi:cm5-lite
----

Unknown tokens used in `implies:` or in `X-Env-Layer-Provides` that are absent from the registry will throw an error at load time. Define all tokens before referencing them.

== Declaring Capabilities on Layers

[[_provides]]
=== X-Env-Layer-Provides

Capability tokens are declared in `X-Env-Layer-Provides` alongside plain label tokens. The engine uses the colon discriminator to distinguish them:

[source,yaml]
----
# X-Env-Layer-Name: rpi5
# X-Env-Layer-Category: device
# X-Env-Layer-Desc: Raspberry Pi 5 device layer
# X-Env-Layer-Version: 2.0.1
# X-Env-Layer-Requires: rpi-device-base,rpi-linux-2712
# X-Env-Layer-Provides: rpi-device,hw:device:rpi:pi5
----

Declaring `hw:device:rpi:pi5` causes the engine to resolve all `implies:` chains and hierarchical parents for that token, injecting the full expanded set into the provider index.

=== X-Env-Layer-RequiresProvider

Declares that a named capability must be present somewhere in the build. The engine validates its existence but imposes no ordering constraint. Use this when a layer is semantically tied to a hardware capability:

[source,yaml]
----
# X-Env-Layer-RequiresProvider: hw:bluetooth
----

The build is rejected at validation time if no layer in the build (and no config override) provides `hw:bluetooth`.

=== AfterProvider restriction

Capability tokens are not permitted in `X-Env-Layer-AfterProvider`. Capabilities express presence - they have no position in the build sequence. The engine rejects capability tokens in `AfterProvider` at parse time with a clear error. Use `RequiresProvider` instead.

== Capability Presence Checks: `has()`

`has('token')` returns `true` if the named token is present in the fully-expanded provider index. `not has('token')` returns `true` if it is absent. Both forms are valid anywhere a conditional expression is accepted.

`has()` resolves against the expanded provider index, so `has('hw:wlan')` is true whether the device declared `hw:wlan` directly, or whether it was inferred because the device declared `hw:wlan:broadcom`, or because `hw:wlan` appeared in an `implies:` chain. The evaluator sees the complete expanded set.

`has()` also works with label tokens. `has('systemd')` returns true if any layer in the build declared `systemd` in `X-Env-Layer-Provides`. Label tokens are indexed alongside capability tokens in the same provider index.

`has()` is an engine metadata-only construct. It is not available in hooks, scripts, or any shell context.

=== In trigger `when=` clauses

Select a LUKS cipher based on hardware crypto availability:

[source,yaml]
----
# X-Env-Var-luks2_cipher-Triggers:
#  when=has('hw:crypto:aes-accel') set IGconf_device_luks2_cipher=aes-xts-plain64 policy=lazy
#  when=not has('hw:crypto:aes-accel') set IGconf_device_luks2_cipher=xchacha12,aes-adiantum-plain64 policy=lazy
----

Layer authors do not need to enumerate device class lists. For example, adding a new capable SoC only requires a new registry entry with the appropriate `implies:` to define the supported feature(s). This means no changes to existing layers would be required.

=== In conflict expressions

Catch an inconsistency between a variant variable and a capability:

[source,yaml]
----
# X-Env-Var-variant-Conflicts: IGconf_device_variant == 'lite' and has('hw:emmc')
----

This fires if a device build configuration specifies a lite variant (eg, a CM5 lite which has no onboard eMMC) and includes a layer that declares `hw:emmc` without the config suppressing it. Conflict expressions are evaluated after variable resolution and after the provider index is built, so both `IGconf_*` values and `has()` are available.

=== In conditional layer dependencies

Pull a layer into the dependency chain only when its required capability is present:

[source,yaml]
----
# X-Env-Layer-Requires: bt-firmware when=has('hw:bluetooth'),rpi-eeprom when=has('hw:eeprom:rpi')
----

DEB822 continuation syntax supported by the layer metadata (indent with a single space) is equivalent, and often more readable for multi-entry lists:

[source,yaml]
----
# X-Env-Layer-Requires: bt-firmware when=has('hw:bluetooth')
#  rpi-eeprom when=has('hw:eeprom:rpi')
----

Each entry may have its own independent condition. Unconditional and conditional entries may be mixed freely.

[IMPORTANT]
====
Conditional requires are evaluated in a single pass against the provider index built from unconditional layers only. A condition that gates on a token provided by another conditionally-pulled layer will not fire - the token is absent from the index at evaluation time. Gate conditional requires only on tokens you know are provided by unconditionally-included layers in your build.
====

== Build Config Overrides

The `capability:` top-level key in a build config YAML is the single override point. It takes precedence over anything declared by the registry or by layers.

[source,yaml]
----
capability:
  hw:bluetooth: false    # carrier board omits BT antenna
  hw:acme:can: true      # carrier board adds a custom CAN interface
----

`false`:: Removes the named token from the provider index meaning that any layer with a `RequiresProvider` on that token will fail validation. If using `has('hw:bluetooth')`, the condition evaluates to false. The suppression applies only to the named token - it does not cascade to child tokens or to tokens that would have been implied by the removed token. Suppress each unwanted token individually.

`true`:: Injects the named token as a provider, the same as if a layer had declared it. For example, use this for custom carrier board hardware that has no dedicated layer.

This mechanism is the canonical way to represent carrier board customisations. `capability: hw:bluetooth: false` is a documented policy decision visible in the config file and not a silent omission.

== CLI: `config --cap`

List all tokens in the loaded registry:

[source,bash]
----
rpi-image-gen config --cap
----

Expand a single token to show its full resolved set (the token itself plus all parents and `implies:` descendants):

[source,bash]
----
rpi-image-gen config --cap hw:soc:bcm2712
----

To include tokens from a custom SRCROOT, pass `-S`:

[source,bash]
----
rpi-image-gen config -S ./my-board --cap
----
